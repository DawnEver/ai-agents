# 在 Claude Code 里协调多个模型,该"切会话"还是"派任务"?

我先把结论放前面。如果你已经在 Claude Code 里干活,经常觉得"这个子任务换个模型会更好",但又不想离开当前会话——那你要的大概率不是"把整个会话切到另一个模型",而是"按任务把活派出去,还能记住上下文"。fabric 就是干这个的,装法一行:`/plugin install fabric@cc-market`。

要不要用,取决于你到底是哪种需求。下面我把两条路的差别摊开讲。

## 为什么"换个模型"这件事一直别扭

我自己用 Claude Code 最常撞见的,是这么一个瞬间:Claude(Opus 4.8 这类顶尖模型)配上 Claude Code,做总控、啃复杂活儿是一等一的,可不是每样活都值得它亲自上。一堆走量的粗活,扔给又便宜、吞吐又大的 DeepSeek 更实在,哪怕它没那么聪明;要出图、改图,Codex 带原生的图像生成、模型底子也不弱,偏偏它那个官方 CLI 用起来挺别扭。模型各有各的最佳位置,却各自为政。

以前解决这个别扭只有两种姿势。一种是"切会话"——把当前对话整体重定向到另一个模型,相当于换了个脑子接着聊。问题是它一次只能有一个脑子,而我要的是让几个模型在同一盘活里分工,不是轮流上场。

另一种是"一次性交接":把子任务甩给另一个模型,拿回结果。方向是对的,可早期我自己写的 takeover 有个硬伤——每次交接都是一锤子买卖,子模型答完就忘,下一轮想追问,上下文得整个重讲一遍。派发于是变成了抽奖:运气好一次问清,运气差就来回补背景。

## fabric 到底是什么

fabric 这名字取的就是"编织"的意思——把这几个模型的调用编织进同一条工作流。它给任何 agent(claude 或 codex 都行)提供一个共享层:在不离开 Claude Code 的前提下,调用、编排、并且和任意 provider 的模型保持多轮会话。

具体我做了 8 个 MCP 工具、5 种调用模式(task / review / agent / image-generate / image-edit),对接三家 provider:claude 走原生 OAuth,codex 走 app-server(负责 review、出图、写入),deepseek 通过 Foundry 环境走 API。provider 列表是动态的,从配置里读,不写死。

## 核心机制:它凭什么能记住上下文

fabric 里我最想讲清楚的一件事,是持久会话——它解决的正是"交接即失忆"那个硬伤。

关键设计我把它压成一句话:**服务器本身就是守护进程**。传统做法要记住多轮状态,得单起一个常驻的 daemon 进程去存会话。fabric 不这么干。MCP server 在 Claude Code 里本来就是个长期存活的进程,那我直接在这个进程内部维护一张会话注册表——一个进程内的 Map,存着每个子会话的活对象。会话的记忆跟着这个 server 一起活着,不需要任何额外的守护进程。

用起来就是四个工具:

```
spawn_session(provider: "deepseek")      → 拿到 sid
session_send(sid, "记住:暗号是 Plumerge")
session_send(sid, "暗号是什么?")          → "Plumerge"
session_close(sid)
```

我真拿这个流程验过:第一轮告诉子模型"暗号是 Plumerge",第二轮它准确复述了出来。对比一下,一次性 call 每次都重开一个干净的子进程,天然记不住——这也是为什么持久会话才是这次真正的升级,而不是又一个 feature。

fabric 不是凭空冒出来的,是 takeover 一路长成的。takeover v1 就一段 Bash heredoc,把 prompt 塞给另一个模型换个结果;v2 长成一个 MCP server,能直接调 Codex、有了几种模式,可它只会一次性派活,几个模型之间彼此无记忆。我想让它们真的"一起"干活,于是加了持久会话、把多家 provider 收敛成一个统一的调用入口——长到这一步,它做的已经不只是"交接任务",而是在编织多个模型的工作流,索性改名叫 fabric。

## 横向对比:三条路差在哪

| 维度 | fabric | 切会话类方案 | 一次性交接(handoff) |
|------|--------|------------|------------------|
| 粒度 | task-level 按任务派发 | session-level 整会话切换 | task-level 单次派发 |
| 多轮记忆 | 持久会话,跨轮记住 | 沿用当前会话历史 | 每次交接即忘 |
| 并发多模型 | 一个终端指挥一支团队 | 同一时刻只有一个模型 | 可多发,但彼此无状态 |
| 依赖 | 零运行时依赖 | 视具体工具 | 视具体工具 |
| 适用 | 编排多模型分工 | 单纯想换个脑子接着聊 | 一锤子买卖的子任务 |

怎么判断该用哪条,我的标准只有一个问题:你是想"换个脑子",还是想"指挥一支队伍"。如果只是当前这段活想让另一个模型接手,切会话足够,也更轻。可只要你需要几个模型在同一盘活里分工、而且分出去的活得记得住上文,前两条路都会卡住,这才轮到 fabric。

## 用下来,几个我自己在意的地方

先说工程可信度,因为这直接决定我敢不敢天天挂着它。fabric 的整个 engine 层大概两千行,从 takeover v2 的三百来行长起来的,但到今天还是零 npm 依赖——纯 `node:` 内置模块。对一个天天跑在我主力会话里的东西,依赖越少,我越睡得着;跨平台也是同理,不引第三方就少一大半平台差异的坑。

再说一个容易被忽略的点:observe-proxy。它不是 MCP 工具,而是一个可选项、也是一个库导出,作用是把子会话的 API 流量镜像一份到磁盘,方便排查。我把它单拎出来说,是因为不少人会误以为它是个"工具"去调——它不是,别找错地方。

至于稳定性,我这套东西是自己吃自己狗粮长起来的:Codex 的一个 echo bug 就是靠 dogfooding 抓出来的(call 返回 `<prompt echo>12`,修成 `12`),测试门从 825 涨到 966 全过、零失败,evolve 循环连着五轮关掉了 38 条 sharp-review 的问题。这些数字不保证没 bug,但至少说明它不是个玩具原型。

## 什么情况下别用它

我得说,它不是给所有人的。如果你平时就用一个模型、也没有"这段活换个脑子更好"的诉求,那 fabric 对你是纯增复杂度,不如不装。

在我看来,如果你要的只是"把整个会话重定向到另一个模型"这种最简单的切换,那些更轻的 session-swap 工具反而更合适,fabric 的会话编排能力在这种场景里根本用不上。

还有一点我得说实话:多 provider 意味着多套凭证和环境。deepseek 要配 Foundry 环境,codex 要能起 app-server。你要是只挂 claude 一家,那持久会话这些能力多半发挥不出来,收益撑不起配置成本。

我回过头看,fabric 想做的其实就一件事:让"换个模型"从"要么切走整个会话、要么交接完就失忆"这两个都别扭的选项里跳出来——按任务派发,还记得住上文。你是不是需要它,取决于你桌上到底是一个脑子,还是一支队伍。

补一句边界:这篇讲的是 fabric 对普通用户最友好的那一面——一行装、按任务派发,小白也能接进来用。它的上限要高得多:我另一篇《让一个 Claude 去盯另一个 Claude》就是拿它的 observe-proxy 去做 Claude Code 运行时观测的进阶用法。想入门从这篇走,想深挖顺着那篇走。

代码在 GitHub,搜 DawnEver/cc-market,装一行 `/plugin install fabric@cc-market` 就能接进你的 Claude Code。
