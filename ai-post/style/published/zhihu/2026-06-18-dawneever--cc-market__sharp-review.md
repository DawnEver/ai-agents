# 从单模型 Code Review 到多模型锐评：sharp-review 的设计取舍与对比


先说一个反直觉的结论：让几个 AI 审同一份代码，最有价值的不是它们都点头的地方，而是它们打架的地方——分歧处，往往就是真 bug 藏身的地方。

顺着这个判断，如果你已经在用 Claude Code，每次写完一个 feature 想找点靠谱的 review 意见，那 sharp-review 值得装。它不跟 CodeRabbit、Greptile 这类 SaaS PR bot 抢同一个位置——那些跑在 PR 上、用单一厂商模型；它在你的 agent loop 里、PR 之前就跑，让 DeepSeek、Claude、Codex 三家不同厂商的模型各从自己的视角独立锐评同一份改动。

sharp-review 是我自己在 Claude Code 上搭的一套插件里负责代码审查的那一环。下面我把它的机制和取舍摊开讲，包括它在哪些场景下不如别的方案。

## 我为什么不信单个模型给我 review

我现在写代码几乎全程挂着 Agent。但 review 这一步，我越来越不敢只看一家模型的意见。

原因很具体。同一个模型连续给自己写的代码挑刺，挑不出几条——它结构上不擅长发现的东西，换一遍上下文还是发现不了。这不是 prompt 写得不够好的问题，是模型本身的盲区固定在那里。WorkOS 那篇用 Cursor Bugbot 复查 Claude Code PR 的文章把这点说得很直白：不同模型抓不同的错。我自己印象最深的一次，是 DeepSeek 盯出了一个 Claude 整段读过却放过的 off-by-one——不是谁更聪明，是它们看走眼的地方根本不在一处。Claude 强在意图理解和长上下文，DeepSeek 便宜、边缘情况上偶尔更敏感，Codex 在工具协调上更稳。它们的失败模式不一样，这才是让它们各看各的有意义的前提。

还有个让我难受的数字：84% 的开发者已经在工作流里用 AI，但只有大约三分之一真正信任它的输出。信任为什么塌？很大程度上是噪声——一个 review 工具如果大部分意见都是废话，你很快就会忽略它，它就成了负资产。所以我想要的不是"更多意见"，是"分歧处的少数意见"加上"≥2 模型都点头的高置信意见"。

## 它到底是什么

sharp-review、rem、takeover 是我在 Claude Code 上做的一组互相打通的插件，sharp-review 是其中负责代码审查的一环，定位是 feature 写完后的"锐评"。一个 Stop hook 在合适的时机触发，挑 3 个里的 2 个不同厂商模型并行 review 你的 diff，把各家结果合并去重、打置信度，最后落成一条当天的记忆文件，同步进 todo 系统。全程本地、Node、几乎零重依赖。

它软耦合了同一套件里的另外两个插件：takeover 负责多模型路由，rem 负责记忆和任务。三者通过一份 deep-merge 的共享状态文件互通，但各自独立——任何一个单独拿出来也能用。

## 它到底怎么让三个模型各看各的

这一节是重点，我把几个关键机制拆开讲，你看完应该能跟同事复述清楚设计。

**多模型 fanout 落到统一 schema。** 三个默认 reviewer 是写死的：一个走 Codex、一个走 DeepSeek、一个走 Opus。每个 reviewer 各拼一段 prompt，通过 takeover 路由到对应厂商，跑完再把模型吐出来的自由文本逼回一个固定的 JSON 结构——severity / file / summary / category / status / suggestion，一条 finding 长这样：

```json
{ "severity": "high", "file": "auth.js",
  "summary": "token 过期未刷新", "category": "logic",
  "status": "open", "suggestion": "在拦截器里补刷新逻辑" }
```

关键在这里：每家模型是**各自独立**跑的，prompt 里没有"参考另一个模型的意见"这一说。它们不商量，各看各的。

**这一步能成立，靠的是 Claude Code 最近更新的 workflow 能力（结构化输出）。** 让三家来源、风格、啰嗦程度都不一样的模型，最后都吐出同一套字段、同一种 severity 取值，这件事过去得靠自己写一堆正则和容错去硬解析自由文本，又脆又难维护。现在 Claude Code 的 workflow 提供了 StructuredOutput 这层平台能力，由它在调用侧把每家模型的输出强制约束到统一 schema——正是因为有了这层约束，后面的合并、去重、置信度排序才谈得上，否则字段对不齐，连"两家说的是不是同一个问题"都没法判断。这套插件能这么做，本质上是借了 Claude Code 平台升级的力。

**2-of-3 轮换。** 每次只跑 2 个，不是 3 个全上——省钱也省噪声。挑哪两个由一个时间种子决定：种子取"距离 epoch 的分钟数"，对组合数取模，在三对组合之间轮。我特意用分钟级种子而不是天级，是因为同一天里你可能 review 好几轮，天级种子会让这几轮都用同一对模型，那轮换就失效了。

**分歧→合并去重→≥2 置信度。** 两家结果回来后，按 file + summary（各取小写前 60 字符拼成 key）分组。同一个问题被 ≥2 个 reviewer 各自独立提到，就标成"高置信（≥2 reviewers）"；只有一家提到的，标"单 reviewer"，但**不丢弃**。这是我和那些"议会/辩论到共识"插件最核心的分歧——后面对比表会展开。


**wave gate 累积阈值抗噪。** 这是我花最多心思防"alert fatigue"的地方。它不是每次会话停下来就 review，而是按**累积改动量**开闸：wave 0（新 commit、新地盘）阈值低，300 行或 5 文件就触发，早点抓问题；wave 1+（同一个 ref 已经 review 过）阈值抬到 1000 行或 15 文件，必须攒够实质改动才再触发。同一个 ref 时只算"自上次 review 以来的增量"，所以你 review 完再加一个文件不会立刻又被触发。纯文档改动直接跳过。HEAD 一移到新 commit，wave 归零。

**rem 记忆 + todo 同步。** review 结果不是聊天框里滚走的一段话，而是写成一条带 frontmatter 的结构化记忆，按日期归档，每条 finding 有 SR-YYYYMMDD-NNN 的稳定 ID。todo 命令直接扫记忆目录出报告，把某条标 fixed，状态就从 open 翻成 fixed。没有会漂移的待办清单。

**参数化引擎。** 同一个 orchestrator 不止能 review 代码。传一个"内容审查"模式，加上自定义 reviewers 和 schema，它就能锐评任意文本。我现在这套 ai-post 的"三方会审"流水线，用的就是同一个引擎，只是把 reviewer 身份换成了"读者代理人"和"技术核查员"。

## 横向对比

把它放进真实的竞争格局里，跟三类方案比，挑的维度都是你真要做决策时会问的：

| 维度 | sharp-review | 单厂商 PR bot（CodeRabbit / Greptile） | consensus 议会 / 对抗辩论插件 |
|------|--------------|--------------------------------------|------------------------------|
| 模型来源 | 跨厂商（DeepSeek + Claude + Codex），各自独立 | 单一厂商单模型 | 多模型，但多为 Claude+Codex+Gemini 同质西方栈 |
| 分歧怎么处理 | 分歧即信号，少数派发现保留；≥2 模型才标高置信 | 无分歧维度（只有一个声音） | 辩论到共识——少数派真发现可能被洗掉 |
| 运行位置 | 本地、agent loop 内、PR 之前 | 云端 SaaS、PR 网关 | 多为本地插件 |
| 输出约束 | 借 Claude Code workflow 的结构化输出，逼各家落到统一 schema | 厂商各自的评论格式 | 多为自由文本，字段不统一 |
| 噪声控制 | wave gate 累积阈值 + 智能过滤 + ≥2 置信度 | 厂商各异；Greptile 召回高(82%)但误报多(~11/run) | 普遍不处理 alert fatigue |
| 结果持久化 | 结构化记忆 + todo 同步，稳定 ID | PR 评论里 | 多为聊天输出，易丢 |
| 成本 | DeepSeek 约 Opus 的 1/50，便宜第三票 | 订阅制 | 全 Claude/Codex/Gemini，偏贵 |
| 适用场景 | 本地写完即审、要跨视角抓盲区 | 团队 PR 流程标准化、要工作流广度 | 要一个"拍板"的统一结论 |

要我说一句最该记住的：**并行不等于独立，议会不等于保留分歧。** 官方那个 code-review 插件是 5 个 agent 并行，但都是同一个模型——5 个一样的脑子共享同一套盲区，并行只是变快，没变更全。而那些辩论到 consensus 的议会插件方向相反：它们让模型吵到达成一致，输出一个共识结论——问题是，共识天然会把"只有一家发现、但其实是对的"那条少数派意见洗掉。

这不是我拍脑袋。arXiv 上关于多模型集成的研究（2502.16279 多模型投票验证、2602.18492 多 LLM 陪审团）反复指向同一点：ensemble 的价值在于**视角的多样性**，让人去比较不同的推理，而不是合成一个单一共识。sharp-review 就是把这条按工程做出来——用 ≥2 一致做置信度排序，同时让单模型的少数派发现活着。

## 用了几天，说几个地方

最直接的体感是噪声真的低。以前我手动让一个模型 review，十条意见里有六七条是"建议加注释""可以考虑提取函数"这种正确的废话。现在两家独立跑下来，被合并标了高置信的那几条，命中率明显高——因为能让两个不同厂商的模型各自独立想到的，通常不是套话。

也踩到过坑。有一次两个 reviewer 只成功回了一个——另一家 takeover 调用失败了，按设计 fallback 成空 findings，结果那次等于退化成单模型 review，置信度那一栏全是"单 reviewer"，整次锐评的价值掉了一截。记忆文件里会写明"only 1/2 reviewers succeeded"，但你得自己留意这行 warning。多模型的好处建立在"多模型真的都跑通了"上，这是个隐性前提。

还有 wave gate 的阈值是按通用项目调的（300/1000 行）。我自己有个改动很碎的项目，老是攒不到阈值，半天不触发——我都开始怀疑是不是它嫌弃我写得太少了；也有过单次重构上千行、一次性塞太多 finding 的情况。好在阈值能按项目调，但默认值不会对每个人都刚好。

## 不适合你的情况

如果你的团队 review 完全跑在 GitHub PR 流程上、要的是评论落到 PR 上、跨平台（GitLab/Bitbucket）打通、和 linter 集成这些工作流广度，那 CodeRabbit 这类成熟 SaaS 比它合适——sharp-review 是本地 in-loop 的，不是 PR 网关。

如果你要的是一个能"拍板"的统一结论、不想看到三个模型各说各话，那 consensus 议会插件更对你的胃口——sharp-review 故意把分歧端到你面前，让你自己判断，这对有些人是负担不是功能。

如果你的项目只接受单一厂商（合规、数据出境限制），那"跨厂商"这个核心卖点直接没了，它退化成一个还不错的单模型 review，性价比不如专门工具。

还有，它依赖 takeover 能把三家模型都调通。哪家厂商当天抽风，那一票就哑了——前面说过的退化问题，这是多模型方案绕不开的脆弱点。

回过头看，sharp-review 想做的其实只有一件事：别让同一个脑子审自己的代码，让几家不同厂商的模型各看各的，把它们的分歧当成信号而不是噪声。它能把每家模型的自由输出逼进同一套结构，也是借了 Claude Code workflow 那层结构化输出的力。如果这正好是你 review 时缺的那块，而且你已经在用 Claude Code，那装一个试试，反正本地跑、DeepSeek 那票还便宜。
