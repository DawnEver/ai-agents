# 深度拆解evolve:一个自迭代插件,和它背后那道注意力保护层

我做了个能自我迭代的插件,但它真正想保护的不是代码,是我。

它表面上是个 review→fix loop,能自己锐评、自己改、跑一轮又一轮。可我做它的时候,心里盯着的从来不是"代码能不能自动变好",而是另一个更要命的问题:当 AI 能同时跑十个 agent loop、我一次只能盯一件事的时候,谁来保护那个盯着它们的人。


## 为什么我先盯上的是注意力,而不是速度

让我先说说做这个东西的起点。

我现在的日常是这样的:Claude Code 能并行帮我跑很多任务,锐评、改 bug、写实现,一口气铺开十个。听起来很爽。但真坐下来用一阵子,我发现爽的是 AI,累的是我。

有一次我同时盯着三个 loop:一个在重构、一个在补测试、一个在改一段我没太看懂的旧逻辑。它们各自跑得飞快,问题是它们都在等我——这个改完了问我对不对,那个测试红了问我要不要回滚,第三个根本不确定它动的那块该不该动。三个窗口轮流闪,我脑子里那根线一次只能接一根。不管 AI 铺开多少个 loop,**到底发生了什么、这改动对不对、先处理哪个、算不算真修好了,最后全压在我一个人身上。**

所以我得到一个有点反直觉的结论:把执行自动化,并没有把我解放出来,只是把我的负担从"自己动手"挪成了"同时监督 N 个 agent loop"。我手里能并行的东西多了,但我这颗脑袋还是一次只能认真看一件事。瓶颈从"AI 能不能干活"变成了"我这一个人的注意力够不够看住这一堆活"。

这个判断,是 evolve 整个设计的地基。我做它,不是为了让 AI 更快,而是为了让 AI 学会替我省注意力。

## 行业都在优化 AI 的速度,没人管那个看着它们的人

我做之前先看了一圈别人在做什么,结论挺一致的。

这一年里,大家围绕"人是瓶颈"这个共识,堆出了一大批工具:Vibe Kanban、Conductor、Claude Squad、各种 agent 看板,Claude Code 自己也上了 Agent Teams,给每个 agent 独立上下文窗口。方向高度一致——**让你能跑更多 agent**。

拿 Vibe Kanban 来说,我真用过一阵。它确实好看:每个 agent 一张卡片,卡住的、改错文件的、能合的,一目了然。可用着用着我发现一件事——它让我"看得更清"了,却没让我"少看"。十个 agent 就是十张卡片整整齐齐排在那,每一张最后还是要我点一下"合"或者"不合"。看板优化的是 AI 这一侧的吞吐,那个坐在看板前要为每一次合并拍板的人,负担一点没轻——我只是从"看一个终端"升级成了"看十张卡片"。

人在环里的疲劳,现在已经不只是体验问题了。我翻到的几篇讨论审批门设计的文章,说得都挺直接:当确认请求来得太勤,人就会停止阅读,养成一种不看内容直接点同意的反射。我自己点 Vibe Kanban 的"合并"点到后来,确实就是这个状态——眼睛扫都不扫,手指自动点同意,点到麻木。这些讨论给的最强解法是**风险分级**:别让每一个低风险的只读查询都来要一次确认,否则你就是在训练审核者闭眼点同意,真正高风险的请求也会吃到同一个反射性的点击。

道理我都认同。但我注意到一件事:这些讨论里,风险分级始终停在原则层面——"可逆性""影响面"是漂亮的概念,却没人把它落成一段你能直接调用的代码。而那些做工具的,要么给你更多看板,要么在某个框架里塞一个通用审批门。**没有人把它做成一道前置在每个执行技能前面、专门保护人注意力的层**。

## 我的解法:在每个技能前面挡一道注意力闸门

所以我换了个角度。我没有再加一个执行技能,而是加了一道**注意力保护层**——一个所有技能都要先过的闸门。


evolve 一轮的执行骨架其实很常规:先锐评,跑至少两个并行审稿人,出一份达成共识、按严重度排序的问题清单;再把这些问题切成互不相交的文件集,分给多个 agent 并行修,保证没有两个 agent 碰同一个文件;然后过 TDD gate,测试全绿才提交,而且是逐条做针对性复查——全局测试绿只是必要不充分,一个声称修好其实没真闭合的问题会被打回 OPEN;最后 scoped commit,逐文件 git add,绝不 -A,你手头无关的工作不会被动到。

这套结构跑起来,我确实不用自己动手了。这种"act → observe → reason → 到条件就停"的写法,跟最近常被提到的 **loop engineering** 是一个路子:你不再亲手 prompt agent,而是去设计那个替你 prompt 它的系统——它要有明确目标、工具、上下文管理、终止逻辑、错误处理。这些 evolve 都有:`checkTermination` 是终止逻辑,rem 负责持久化,TDD gate 兜错误处理。但 loop engineering 里有一条最关键、也最常被一笔带过的:这个自动跑的系统,到底该在什么时候、用什么方式把控制权交回给人。骨架我讲完了,真正的重心在末尾那道闸门——它就是我对这个问题给出的答案。

## 这道闸门到底怎么决定要不要打扰我

这道闸门的全部职责,就一句话:**没有任何技能可以直接消耗我的注意力,它必须先通过这道闸门。**

闸门做三件事,我一件件说。

**第一步,先分类:这事要不要惊动人。** 它把每个待决项分成两堆——要我拍板的,和可以自动放行的。放行的判据我写得很死:可逆,有安全默认值,且不是 HIGH 风险。三个条件得同时成立。这其实就是把那个抽象的"风险分级"落成了一段大概三十行的纯函数,可逆性、影响面、置信度三个维度,直接编码成一个布尔判断。原则不是我发明的,我只是把它写成了可执行、可测试、零依赖的代码。

**第二步,真要惊动我时,只给我该看的。** 那些确实要我决定的,它不会把原始发现一股脑甩过来。它压成固定四个字段:

```
mustKnow              我此刻必须知道的
decision              我必须做的决定
consequenceIfIgnored  如果我不管,后果是什么(可逆 / 不可逆)
defaultIfIgnored      我什么都不说时,会落到哪个安全默认
```

这四行几乎是我那条原则的逐字翻译。一个发现被压成这样,我扫一眼就能拍板,不用再去翻它从哪来、上下文是什么。

**第三步,别让我点十次。** 剩下这些真要我决定的,它不一个一个弹窗,而是合并成一个提示:最多四个问题,最高风险的排前面,多出来的溢出到下一轮。我一次面对的,永远是一个被压缩、被排序、被合并过的决策面。

三条全中,自动放行。否则,留给我。

如果调用方是另一个 AI(headless、被上层编排器调起来的场景),这道闸门走另一条路:从不弹窗,按策略自动决议,只把那些不可逆、又没有安全默认的留着不动(标 OPEN 记日志),但它永远不阻塞。

## defer ≠ drop:别打扰我,不等于悄悄丢掉

讲到这我得停一下,因为这里有个我特别在意的点。

"别打扰我"最危险的滑坡,是滑成"悄悄丢掉"。前面那些讲疲劳的讨论也警告过:被压下去的告警容易被丢。所以我给闸门加了一条硬保证——**defer ≠ drop**。

一个被延后的发现(哪怕是 LOW),会一直留在待办里保持 OPEN 状态;下一轮 fan-out 不管它什么风险级别,都会**再尝试修一次**;如果它卡了三轮还没修掉,卡死计数触发,这道闸门会主动把它升级到我面前。

换句话说,这道闸门管的永远是**要不要打扰你**,从不是**要不要修它**。这一条,是我跟"通用审批门"最不一样的地方:大多数审批门只决定弹不弹,弹过没人理的就沉了;我这道闸门把"不弹"和"一定会被继续修"绑死成了一对。

## 它是一套框架的一环,不是一个孤立的插件

这里要补一句,否则容易把 evolve 看小了。evolve 自己并不全能,它硬依赖我之前做的另外三个插件,缺一个就跑不起来,我也故意没留 fallback:**sharp-review**(我做的锐评引擎)负责"找"(每轮锐评就是它跑的,≥2 reviewer、稳定 SR 编号)、**rem** 负责"记"(loop 状态 `evolveState` 和每轮 round-log 都存在 rem 里,跨轮跨会话的连续性靠它)、**todo** 负责"追"(finding 状态以 todo / sharp-review.md 为事实源)。evolve 本质上是个编排器:找、修、记、追咬合成一条完整流水线。

而真正让它成为"框架"而非"几个各跑各的脚本"的,是注意力闸门所在的位置——它不在 evolve 里,它在 `shared/`,被打包进**每一个**插件。也就是说这道闸从第一天起就不是 evolve 的私有功能,而是整套框架的公共保护层:任何一个会消耗我注意力的插件,走的都是同一道闸、同一套 classify / compress / route、同一条 defer ≠ drop。执行技能可以一个个加,但保护我注意力的方式全框架只有一套。这也是我跟"在某个框架里塞一个审批门"最不同的地方:我不是给某个流程加确认,我是给整套系统加了一层统一的注意力前置层。

## 横向对比

下面这三列,代表三条不同的路线:我的注意力闸门、Agent 看板那套可视化调度、还有框架内的朴素审批门。带着这个坐标进表会清楚很多。

| 维度 | evolve 的注意力闸门 | Agent 看板(Vibe Kanban / Conductor 类) | 框架内通用审批门(LangGraph interrupt 类) |
|------|------|------|------|
| 解决的瓶颈 | 人的注意力带宽 | 多 agent 的可视化调度 | 单次动作的人工确认 |
| 形态 | 前置在每个技能前的复用层 | 看板 / 仪表盘 | 框架内的一个中断原语 |
| 优化对象 | 那个看着 AI 的人 | AI 这一侧的吞吐 | 单个动作的安全 |
| 风险分级 | 落成 ~30 行纯函数(可逆 / 影响面 / 置信度) | 基本没有,靠人看 | 有钩子,但要你自己写策略 |
| 打扰的处理 | 自动放行可逆项 + 压成四字段 + 合并成一个提示 | 全摊在看板上等你扫 | 命中即同步打断 |
| 漏掉的保证 | defer ≠ drop,延后项每轮重试 + 卡死升级 | 无,卡片可能一直堆着 | 无,被否决/忽略的就过去了 |
| 适合谁 | 一个人盯 N 个自治 AI Agent 的一线开发 | 团队 / 多人协作看进度 | 在框架里搭 HITL 流程的人 |

这张表我想说的就一件事:别的方案大多在优化 AI 怎么跑得更快、或者怎么让你看得更清,而我这道闸门唯一在意的,是那个看着它们的人,他的注意力够不够花。看板让你看得更清,但它没替你省注意力——它只是把十个打扰整整齐齐摆出来。

## 用下来,一个我自己也没料到的点

真正写出来用了一阵,有个地方跟我预想的不太一样。

最意外的是,自动放行的那一堆"可逆+有默认+非 HIGH"的小事,占比比我以为的高得多。一轮跑下来,真正需要我拍板的常常就一两条,剩下的全被默认值悄悄消化了。我原本担心三十行的判据会不会太糙、放过不该放的,但因为判据卡得保守(三个条件全满足才放),实际跑下来误放的情况很少。

## 什么情况下你不该用它

说点不适合的。

如果你是团队协作、要多人看同一批 agent 的进度,那你要的是看板,不是这道闸门——闸门保护的是单个人的注意力,它不解决多人可视化。如果你的工作流里几乎没有可逆的小决策,全是必须人工逐个确认的高风险动作,那自动放行那一层基本用不上,闸门的价值会大打折扣。还有,evolve 硬依赖 sharp-review、rem、todo 三个 cc-market 插件,缺一个就跑不起来,没有 fallback——这是我故意的(我不想维护一堆降级路径),但对你意味着接入成本不低。

它也不是什么银弹。它没有 benchmark,因为它本来就不是个吞吐工具——它优化的是一个没法被基准测试量化的东西:我一天里被打扰了几次。

## 最后

这里有个我自己觉得挺好笑的反讽:Claude Code 这类工具本来是来替我省力的、省注意力的,结果我用着用着,反倒得亲手给它装一道闸,专门拦住它别来耗我的注意力。省力工具用到要给它上保护层,这件事本身就够说明问题了。

evolve 的 slogan 我写的是「加入光荣的进化吧!」。它是个双关。第一层是字面的:这个插件让代码每一轮自我"进化"一点。但我真正想说的是第二层——我想进化的,从来不是逼自己追上 AI 的速度,而是**人机协作的方式本身**:让 AI 学会替我省注意力,把人从监督负担里解放出来,把注意力当成一份要精打细算的预算来管。

如果你也是一个人盯着一堆自治 AI Agent、被确认弹窗追着跑的一线开发,这道思路或许值得你看看:https://github.com/DawnEver/cc-market(`evolve/` 插件)。

说到底,在这个并行 AI 时代,有个数字是任何 benchmark 都测不出来的——我这一天,到底被打扰了几次。
