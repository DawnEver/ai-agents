# sharp-review 架构拆解：我如何把「多模型独立锐评」做成代码审查流水线


写完一个功能，按下回车，Claude 顺手帮我 review 了一遍，三条建议，全是夸我的。我盯着那三条看了半天，心里只有一个念头：它在审自己刚写的代码，我让贼来抓贼。

一个模型有它结构性的盲区——擅长架构的，对边界条件就钝；对工具调用敏感的，又看不出抽象漏了。你让它自己审自己，盲区原封不动地传了下来。哪怕并行跑五个，只要是同一个模型，五份盲区是同一份。

所以我没在「哪个模型最强」上纠结。我换了个思路：找几家不同厂商的模型，DeepSeek、Claude、Codex，让它们各自从自己的视角，独立锐评同一份代码。谁也不商量，谁也不知道别人写了啥。它们看法一致的地方，大概率没事；它们打架的地方——那才是真 bug 藏身的地方。

这个插件我管它叫 **sharp-review**。它是我自己在 Claude Code 上搭的一套插件体系里的一环——sharp-review 管代码审查，rem 管记忆和待办，takeover 管多模型派发，几个插件互相打通。下面我把 sharp-review 的架构整个拆开讲一遍，讲清楚什么时机触发、数据怎么流、为什么这么设计，你看完应该能自己照着搭一套。


## 一份代码，三个互不通气的视角

先说最核心的设计，也是整个插件立身的地方：**分歧即信号**。

我见过不少多模型 review 的方案，思路都是「让几个模型辩论到达成共识」。听起来很美，实际上有个坑——共识会把少数派的真发现洗掉。三个模型里只有一个嗅到了那个并发隐患，另外两个没看出来，一辩论、一投票，这条就被当噪声压下去了。可偏偏它是对的。

我反过来用。我不追求共识，我**盯着分歧**。一个发现被两个以上模型同时提到，标成高置信度；只被一个模型提到的，我不丢，标成 single-reviewer 留着——因为那很可能就是它独有视角抓到的东西。合并的逻辑很朴素：把所有 findings 按关键字段去重归组，然后数一下每组命中了几个 reviewer，命中人数决定置信度。

```
group.length >= 2 → high-confidence (≥2 reviewers)
否则              → single-reviewer（不丢，留着）
```

一行判断，但它背后是整个哲学的转向：我要的是视角的多样性，不是一个被磨平的最终答案。学术上这事也有支撑——arXiv 上关于多模型 ensemble 的研究反复说，集成的价值恰恰在于「不同的解释路径」，而不是合成一个统一结论。我把这句话直接做进了逻辑里。

## 三选二：让模型轮班，而不是全员上工

每次 review 我不会把三个模型全叫起来。默认配置是三家——Codex、DeepSeek、Opus，但每一轮只挑其中两个上工。

为什么不全上？一是成本，三家全跑一遍，token 烧得肉疼；二是我想让组合**轮换**起来，这一轮 Codex+DeepSeek，下一轮 DeepSeek+Opus，长期下来三种视角都能照到我的代码，不会永远是同一对在看。

挑哪两个，我用的是 seed-mod 轮换。seed 取「距 epoch 的分钟数」，这样哪怕同一天里我连审好几轮，每一轮的搭档也会变，而不是死循环同一对。思路很简单：

```
组合数 = n × (n-1) / 2      // 三家 → 三种两两组合
落在哪对 = seed % 组合数      // (0,1)/(1,2)/(2,0) 取一对
```

说白了就是把三种两两组合排成一圈，seed 转一下指针落在哪对就用哪对。无状态，不需要记「上次用了谁」，纯靠时间种子自己转。我特意没存轮换索引——少一个状态文件，少一处能出 bug 的地方。


## 模型怎么落到一张统一的表上

光把模型叫起来还不够，更麻烦的是：三家模型脾气不一样，DeepSeek 爱长篇大论，Codex 爱给你贴代码，Opus 爱讲架构。它们各说各话，我怎么把这堆口水合并成一张能用的表？

答案是 **schema 约束**。我给每个模型套了一个 JSON Schema，逼它把锐评意见塞进固定字段里——severity（严重度）、file（文件）、summary（一句话）、category（Bug/Feature/Performance）、suggestion（一句话修法）。模型怎么想是它的自由，但吐出来必须长这样：

```json
{
  "severity": "HIGH",
  "file": "src/foo.js",
  "summary": "并发写入未加锁，存在竞态",
  "category": "Bug",
  "suggestion": "改成原子操作或加互斥锁"
}
```

这件事能成，得感谢 **Claude Code 最近更新的 workflow 功能**——也就是它新提供的结构化输出（StructuredOutput）能力。它强制每家模型把原本天马行空的自由文本，塞进我指定的同一套 JSON schema 里。没有这个，三家吐回来的就是三种格式的散文，字段都对不齐，谈不上去重和打分；有了它，我才能把多家模型的锐评收拢成一张干净的表。我搭的这套插件逻辑，恰好踩在了平台刚长出来的这块能力上。

模型派发我没自己造轮子，走的是 **takeover**——我那套插件体系里专门干多模型路由的一环。我把每个 reviewer 包成一次 takeover 调用，一个路由到 codex，一个到 deepseek，一个到 opus，然后并行发出去。三家同时锐评，谁先回来不重要，最后我按 schema 把它们的 findings 收齐、去重、数命中、编号（`SR-YYYYMMDD-NNN`），一张干净的表就出来了。

这里有个我踩过的细节：结构化输出的 schema 必须是 `{ "findings": [...] }` 这种对象套数组，不能直接给一个裸数组——给裸数组模型会静默失败，一条都不返回，还不报错。我在设计约束里专门钉了这条，就是因为吃过这个亏。

## 别让 AI 一改文档就来烦你：wave gate

做到这一步，功能是齐了，但很快我就遇到新问题：**它太勤快了**。我改个错别字、动一行注释，它都要郑重其事地审一遍，弹一堆「建议」。审多了我就开始无视它，工具一旦变成噪声，就废了。

这其实是 2026 年 AI code review 最大的痛点——不是漏 bug，是**警报疲劳**。有数据说 84% 的开发者在用 AI，但只有约三分之一信任它的输出。信任就是这么被噪声一点点磨没的。

我的解法是 **wave gate**：review 不按「每次停下」触发，而是按**改动累积量**触发。门槛分两档：

- **wave 0**（新 commit）：改动 ≥300 行 或 ≥5 个文件才触发——早发现早除虫
- **wave 1+**（同一个 ref 已经审过）：要 ≥1000 行 或 ≥15 个文件才再审——审过的别老来烦

没到门槛的那些停顿，改动不会清零，而是**一直攒着**，攒够了再一次性触发。加上一层智能过滤——lockfile、生成产物、二进制、纯改名，这些直接踢掉不算数。文档类改动（`.md`、`.claude/`）更是连门都进不来。


效果就是：它只在我真动了筋骨的时候才开口。其余时间安安静静。一个会闭嘴的工具，我才信得过。

## 同一个引擎，审代码也审文章

讲到这，你可能以为 sharp-review 只能审代码。其实我从一开始就把它做成**参数化的**——审查目标和审查逻辑是解耦的。

整个引擎暴露出一堆参数：谁来审（reviewers）、结果长什么样（findingSchema）、审哪些维度（reviewScope）、按什么去重（dedupKeyFields）、编号前缀（idPrefix），还有一个开关在「审代码」和「审文章」之间切。换句话说，只要我把这几个参数换一套，同一个 fanout→schema 约束→合并去重→置信度的流水线，就能拿去审任意文本。

最好的证据就是你正在读的这篇文章。它出自一条我自己搭的内容流水线，而那条流水线的「三方会审」环节，复用的正是 sharp-review 这台引擎——只不过我把 reviewers 换成了「读者代理人」和「技术核查员」两个身份，把结果维度换成了「AI 味、节奏、humor 密度」这类东西。审代码时我用 seed-mod 三选二；审文章时我换成「所有身份全上」。同一台引擎，两种活。我写它的时候就没打算只让它审代码。

## 锐评完了，别让它随聊天记录飘走

还有最后一个我特别在意的点：大多数 AI review 的结果是飘在聊天记录里的——你往上一滚，它就没了，下次还得重审一遍。

我不接受这个。sharp-review 的每次结果都落成一个结构化的记忆文件，路径形如 `.claude/memory/YYYY/MM/DD/sharp-review.md`，带 rem 的 frontmatter。这文件就是唯一真相源，没有派生的 `tasks.md`。然后它和 **rem** 的 todo 系统打通——`todo` 命令直接扫这些记忆文件，实时生成待办，永远不会过期。修好一条，我敲一句就行：

```bash
todo mark SR-20260617-001 fixed
```

这一下会把那条 finding 的状态从 `OPEN` 翻成 `FIXED`，frontmatter 自动重算。锐评意见不再是聊一聊就忘的废话，而是一条能追踪、能勾销、能跨会话翻出来的账。

## 快速上手

整个插件是 Node.js 写的，没有重依赖。装好之后，在 Claude Code 里手动触发一次锐评：

```bash
/sharp-review
```

不带参数就审你当前未提交的改动（staged + unstaged）。想审某个范围或某个子目录：

```bash
/sharp-review --range "main...HEAD"
/sharp-review --path "src/components"
```

平时你根本不用手动敲——wave gate 挂在 Stop hook 上，改动攒够了它自己触发。审完去看 `todo`，该修的修，修完 `todo mark` 勾掉。就这样。

## 写在最后

回过头看，sharp-review 的三个核心其实是同一个判断的三个侧面：**单一模型有共同盲区，那就让不同厂商的模型各看各的，盯着它们的分歧。** wave gate 让它不烦人，参数化让它能审代码也能审文章，rem+takeover 让它有记忆、有派发、有待办闭环——这几个插件本来就是我在 Claude Code 上一起搭、互相咬合的一套东西。
